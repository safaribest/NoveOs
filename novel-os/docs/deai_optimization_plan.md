## 二、根因诊断

### 2.1 从“人类写作”角度看，AI 味的本质

人类爆款网文的核心不是“没有修辞”，而是：

1. **有明确的声音（Voice）**：同样的情节，不同作者写出完全不同的语感。
2. **有信息差和意外**：读者不知道下一步，但回头看又觉得合理。
3. **有真实的人物**：每个人说话带口癖、有情绪起伏、会犯错。
4. **有呼吸感**：长短段落交错，紧张处短，叙述处长。
5. **有节制**：同一个意象不会每章都出现。

当前系统的问题，是把“人味”当成一套**可枚举的禁止清单**来执行，结果写出的是“没有明显错误的平庸网文”——这正是 AI 味。

### 2.2 系统架构层面的 6 大病根

| 病根 | 表现 | 责任模块 |
|------|------|----------|
| **1. 反检测改写器越改越 AI** | 随机插入碎片化短句、低频词替换、半句话 | `core/anti_detect_reviser.py` |
| **2. 校验器奖励碎片化** | >30 字段落就警告，把正常叙述段切成诗 | `core/chapter_validator.py` |
| **3. Prompt 用负面清单当模板** | 17 条铁律导致模型反向复读“不是X，是Y” | `core/writing/prompts.py`、`book.yaml` |
| **4. 字数硬截断** | 20 章 15 章结尾是半截话 | `core/writing/pipeline.py` |
| **5. 缺少数值化、可执行的规则引擎** | 规则全靠 LLM 自觉，漏判严重 | 无 |
| **6. 前文污染** | 4–20 章旧 AI 味会成为后续章节上下文 | `world_state.db` / pipeline |

### 2.3 内容层面的 4 大病根

| 病根 | 表现 |
|------|------|
| **1. 大纲强制每章“打脸+解锁技能”** | 导致每章结构雷同，数字和设定堆砌 |
| **2. 缺少目标风格示例** | 模型只能背规则，无法模仿真人语感 |
| **3. 人物声线表缺失** | 赵德、周鸣、冷清月说话像一个作者 |
| **4. 感官入口单一** | 每章都回到“虎口旧疤/黑丝蠕动” |

## 三、整体解决方案

### 3.1 架构原则

> **从“禁止清单驱动”转向“声音示例 + 规则引擎 + 多 Agent 审查”三位一体。**

```text
┌─────────────────────────────────────────────────────────────┐
│  Data Layer（声音资产）                                      │
│  - 项目级 voice_examples.md（3–5 段目标风格手造示例）        │
│  - 人物声线卡 voice_cards.md（赵德/周鸣/冷清月/苏晚照）      │
│  - 品类级 style_guide/examples/fantasy.md（玄幻通用示例）    │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Prompt Layer（轻量化约束）                                  │
│  - 只保留 3–5 条核心原则（去情绪标签、去系统面板、去精确参数）│
│  - 用示例替代负面清单                                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Pipeline Layer（生成）                                      │
│  Director → BeatPlanner → StyleRetrieval → SceneWriter      │
│  → HookEngineer → DialogueTuner → StyleCritic → Polish      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Review Layer（多 Agent 审查）                               │
│  - PatternCritic（句式/比喻/面板词/情绪标签）                │
│  - RepetitionCritic（复读意象/同质化开头）                   │
│  - VoiceCritic（段落节奏/对话声线）                          │
│  - ConsistencyCritic（与前章设定/语气一致性）               │
│  - StyleRuleEngine（数值化硬规则兜底）                       │
│  → RevisionPlanner → StyleReviser → Judge（原稿 vs 修改稿） │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Output Layer（后处理）                                      │
│  - 自动截断修复（按完整句末）                                │
│  - 标题提取/补全                                             │
│  - AI 味评分报告                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 核心模块设计

#### 3.2.1 StyleRuleEngine（规则引擎）

**作用**：把“禁止清单”变成可执行、可测试、不会漏判的代码规则。

**检测项**：

| 规则 | 实现方式 | 自动修复 |
|------|----------|----------|
| `不是X，是Y` | 正则 `不是[^，。！？]{1,35}是` | 否（标记） |
| `像……` 比喻 | 正则 `像[^，。！？]{1,25}` | 否（标记） |
| 系统面板词 `【】` | 正则 `【|】` | 是（去除括号） |
| 情绪标签 | 关键词表 | 否（标记） |
| 精确数字参数 | 正则 + 中文数词密度 | 否（标记） |
| 重复意象超限 | 每关键词每章阈值 | 否（标记） |
| 结尾不完整 | 末字符标点检查 | 否（截断修复） |

**为什么放在代码里**：LLM 不擅长计数和严格规则，代码不会漏判。

#### 3.2.2 多 Agent StyleCritic

**当前已实现 3 个 critic，建议扩展到 5 个**：

| Critic | 职责 | 输出 |
|--------|------|------|
| PatternCritic | 句式/比喻/面板词/情绪标签 | JSON issues |
| RepetitionCritic | 复读意象/同质化开头 | JSON issues |
| VoiceCritic | 段落节奏/对话声线 | JSON issues |
| ConsistencyCritic | 与前章设定、人物状态一致性 | JSON issues |
| ToneCritic | 整体语感是否贴近 voice_examples | JSON issues |

**RevisionPlanner**：汇总 issues，去重，生成“最小修改计划”。

**StyleReviser**：按修改计划执行，一次改完，避免反复调用。

**Judge**：用规则引擎分别给原稿和修改稿打分，取高分版本。

#### 3.2.3 Voice Examples（声音资产）

**为什么比规则更重要**：

人类学写作不是背“不要写不是X是Y”，而是读张爱玲、读古龙、读猫腻，然后模仿。LLM 也一样。

**需要制作**：

1. **项目级 `voice_examples.md`**：3–5 段由人类（或严格审定）写成的目标风格段落，直接注入 SceneWriter prompt。
2. **人物声线卡 `voice_cards.md`**：每个人物的口头禅、语速、禁忌词、情绪表达方式。
3. **更新 `core/style_guide/examples/fantasy.md`**：替换当前“斗罗大陆式”示例为冷硬诡异修仙风格示例。

### 3.3 内容层面调整

| 问题 | 解决方案 |
|------|----------|
| 每章强制打脸+解锁技能 | 大纲里允许“过渡章”“受伤章”“纯对话章”，降低结构雷同感 |
| 精确数字过多 | 大纲中的数字在正文里模糊化：三日后 → 过了两天；百年灵草 → 药龄很足 |
| 感官入口单一 | 每章强制换一个身体反应入口（胃袋、后颈、膝盖、眼窝、耳根） |
| 反派功能化 | 给赵德、周鸣加小动作和私心，不只是挑衅工具人 |

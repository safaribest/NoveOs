# Novel-OS 全自动长篇小说写作系统

> **对外宣传版 v1.0**  
> 一个基于自研 Agent 编排引擎的无人值守长篇小说生产系统

---

## 一、项目定位

| 维度 | 说明 |
|------|------|
| **核心目标** | 实现 200 章、40 万字长篇小说的**全自动无人值守写作** |
| **适用场景** | 网文平台连载（番茄/起点等）、IP 孵化、有声书改编 |
| **核心价值** | 将单本小说写作成本从数万元降至 **10-20 元**，周期从数月压缩至 **数天** |
| **当前状态** | 系统运行中，已自动生成 **18 章**，质量稳定（1900-2600字/章） |

## 二、业务流程（端到端）

```
【策划层】
创作方案（世界观/人物/大纲）
    ↓
book_data.py（结构化创作数据：200章大纲 + 7个角色 + 12个技能 + 15条硬规则）
    ↓
【数据层】
init_book.py 初始化 → world_state.db（SQLite 运行时数据库）
    ↓
【写作层】
自研 Agent 编排引擎（逐章调度）
    ↓
Director（任务卡生成）→ Writer（正文写作）→ Polish（润色）→ Auditor（硬指标审计）
    ↓
【产出层】
chapters/第XXX章_标题.txt（200章完整小说正文）
```

### 2.1 关键流程节点

| 节点 | 功能 | 自动化程度 |
|------|------|-----------|
| 大纲设计 | 人类创作者完成世界观、人物、200章细纲 | 人工 |
| 数据初始化 | 一键导入到数据库，自动生成角色出场表、技能解锁表、术语字典 | **全自动** |
| 逐章写作 | Agent 编排引擎按章调用 AI，每章经历"任务卡→写作→润色→审计"四步 | **全自动** |
| 质量检查 | 本地规则引擎检查字数、禁用词、他字密度；每5章批量审阅字数稳定性 | **全自动** |
| 失败重试 | 字数不足或审计挂科时自动重试（最多3次），仍不达标标记为"待精修"继续下一章 | **全自动** |
| 持续运行 | Windows 独立常驻进程，不受终端关闭影响，可持续运行数天 | **全自动** |

## 三、技术架构

### 3.1 四层领域模型

```
┌─────────────────────────────────────────────┐
│  第一层：项目层（Project）                      │
│  book.yaml + book_data.py + outline.md        │
│  单本书的所有创作数据，人类可编辑的源头          │
├─────────────────────────────────────────────┤
│  第二层：核心层（Core）                         │
│  cli.py / init_book.py / config_loader        │
│  命令入口、配置解析、数据导入、状态管理          │
├─────────────────────────────────────────────┤
│  第三层：写作领域层（Writing Domain）            │
│  context.py / pipeline.py / prompts.py        │
│  上下文构建 → 四步流水线 → Prompt 工程         │
├─────────────────────────────────────────────┤
│  第四层：基础设施层（Infrastructure）            │
│  llm_client.py（DeepSeek API）+ SQLite        │
│  LLM 调用封装 + 数据持久化                      │
└─────────────────────────────────────────────┘
```

### 3.2 核心技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 编程语言 | Python 3.12 | 后端全栈 |
| Web 框架 | FastAPI | API 层（可选） |
| 数据库 | SQLite | 轻量、零配置、单文件 |
| LLM | DeepSeek-V3（deepseek-chat） | 中文写作质量最优的商用模型 |
| API 封装 | LiteLLM | 统一多模型调用接口 |
| 进程管理 | Windows 独立进程（wmic） | 脱离终端限制，持续运行 |

### 3.3 自研 Agent 编排引擎

**不是 CrewAI，不是 LangChain，是专为长篇小说写作自研的轻量编排引擎。**

设计哲学：
- **轻量**：只依赖 `litellm` 调 API，无框架依赖地狱
- **可控**：Prompt 和调用逻辑完全透明，可逐行调试
- **稳定**：不会因为第三方框架版本升级而崩溃

引擎结构：
```
外层调度器（outer_crew_runner）
    ├── 每5章：全书架构巡检（偏离分析/角色活跃度/伏笔健康度）
    ├── 每10章：节奏分析（爽点密度/字数趋势/情绪曲线）
    └── 发现矛盾时：回溯修正方案生成

内层流水线（pipeline）
    ├── Director：根据大纲+角色状态生成本章任务卡
    ├── Writer：按任务卡写作2200字正文
    ├── Polish：消除AI味，提升画面感
    └── Auditor：审计字数/他字密度/禁用词/DNA匹配度

精度层（validation）
    ├── ChapterValidator：硬指标检查
    ├── StateManager：跨章状态一致性
    └── Expander：字数不足时自动扩充
```

## 四、核心创新点

### 4.1 数据驱动的创作约束

传统 AI 写作的问题是"越写越崩"——人物性格漂移、技能提前解锁、伏笔忘记回收。

Novel-OS 的解决方案：**所有创作约束写入数据库，每章写作前自动注入 Prompt。**

| 约束类型 | 示例 | 注入方式 |
|---------|------|---------|
| 角色状态 | 陆远舟左膝半月板损伤，每次使用铜鼓震步后剧痛 | 本章角色状态卡片 |
| 技能解锁 | 鼓楼视野=第2章解锁，铜鼓震步=第5章解锁 | 技能树检查，未解锁禁止出现 |
| 术语字典 | 侗神瞳、声浪值、鼓楼视野、榫卯传球等16条 | 术语首次出现标注 |
| 写作规则 | 禁用"首先/其次/最后"、禁用情绪标签、每章至少3次弹幕 | Auditor 硬审计 |
| 伏笔债务 | 膝盖旧伤（第6章埋下）→ 世界杯预选赛被利用（第180章回收） | 债务表自动追踪 |

### 4.2 品类 DNA 匹配度评分

不同平台、不同题材有完全不同的"节奏 DNA"。Novel-OS 内置品类 DNA 库：

- **都市体育+系统流+直播文**（当前项目）
  - 爽点密度：每章至少1次
  - 直播弹幕占比：10-15%
  - 单句长度：≤25字
  - 他字密度：控制在合理范围
  - 情绪循环：压抑→蓄力→爆发，每3-5章一轮

Auditor 每章输出 DNA 匹配度评分，低于阈值自动重试。

### 4.3 侗族文化差异化（示例项目）

在番茄小说平台的体育文赛道中，**侗族文化元素（鼓楼、铜鼓、萨玛、多耶战舞、侗医）几乎是独一份的记忆点。**

系统通过 `term_dict` 表管理所有文化术语：
- 首次出现章节标注
- Director 任务卡自动提示"本章需出现的文化元素"
- 确保文化符号持续在场，不流于表面

## 七、诚实声明

> 我们不夸大技术，以下是我们**没有**做的事：

1. **没有使用 CrewAI 框架** —— 系统是自研的轻量编排引擎，借用了 CrewAI 的多层 Agent 概念，但底层是手动调 DeepSeek API。
2. **没有使用 LangChain** —— 无框架依赖，只用一个 `litellm` 包。
3. **不是"一键生成"** —— 前置需要人类完成世界观、200章大纲、角色设计（约 1-2 天工作量）。
4. **不完美** —— 约 15% 的章节需要 1-2 次重试，约 5% 的章节最终仍不达标需人工精修。

> 我们做的是：**把长篇小说写作中最枯燥、最耗时、最容易出错的"逐章执行"环节，交给一个稳定、可控、低成本的自动化系统。**

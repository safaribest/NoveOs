# Novel-OS 业务结构梳理 v2.0

> 基于 2026-06-13 项目状态：已完成《村超系统》114/200 章，数据库已修复并重新初始化。

---

## 一、业务定位

**Novel-OS 是一个 AI 驱动的长篇网文工业化生产系统**，核心目标：

- **低成本**：单本 10-20 元（主要靠 DeepSeek API）
- **高速度**：单章 1-3 分钟，200 章无人值守
- **可发布**：去 AI 味、人设稳定、节奏符合平台规则
- **差异化**：侗族文化 + 体育系统 + 直播弹幕，形成独特文风

当前聚焦项目：《村超系统：我带侗寨球队踢爆世界杯》（200 章目标）。

## 二、核心业务流程

```
┌─────────────────────────────────────────────────────────────────┐
│                        1. 创作数据准备                            │
│   book_data.py（200章大纲、角色、债务、伏笔、规则、技能）           │
│   book.yaml（项目元信息、作者人格、Agent配置、平台规则）            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                        2. 项目初始化                              │
│   init_book.py → world_state.db（outline/characters/terms/debts...）│
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                        3. 单章/批量写作                           │
│   cli.py write --chapter N                                       │
│   auto_write.py --start N --limit M                              │
│   generate_cilin_chapters.py（前5章实验性辞林生成）                │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│                        4. 章节产出                                │
│   chapters/第XXX章_标题.txt                                       │
│   world_state.db/chapter_history（摘要、字数、情感坐标）            │
└─────────────────────────────────────────────────────────────────┘
```

## 三、数据架构

### 3.1 静态配置层

| 文件 | 作用 | 维护方式 |
|------|------|---------|
| `book.yaml` | 项目元信息、作者人格、Agent LLM 配置、字数目标 | 人工编辑 |
| `book_data.py` | 200 章大纲、角色矩阵、债务/伏笔、写作规则、技能树 | 人工编辑 |
| `templates/config_base.md` | 通用配置模板，渲染后作为 system prompt | 模板 |
| `templates/scene_lexicon.yaml` | 场景辞林：词条、情绪子类、红线 | 人工编辑 |
| `templates/dialog_taboo.yaml` | 对话红线：角色词汇地址、禁用词 | 人工编辑 |

### 3.2 动态状态层

`world_state.db` —— SQLite 多项目数据库：

| 表 | 用途 | 当前数据量 |
|----|------|-----------|
| `projects` | 项目注册与外键根 | 1 |
| `outline` | 200 章大纲（含 title/core_event/hook 等） | 200 |
| `character_states` | 角色动态状态（位置、情感、对话指纹） | 7 |
| `term_dict` | 世界观术语与技能说明 | 17 |
| `debts` | 待回收的叙事债务 | 6 |
| `foreshadowing` | 伏笔总表 | 8 |
| `skill_tree` | 技能解锁进度 | 12 |
| `chapter_history` | 已写章节摘要、字数、标题 | 114 |
| `chapter_metrics` | 章节质量指标（IWR/平台分/DNA 匹配） | 待补充 |
| `genre_dna` | 品类 DNA 基准 | 1 |

### 3.3 产出物层

| 路径 | 说明 |
|------|------|
| `chapters/第XXX章_标题.txt` | 最终正文 |
| `logs/auto_write.log` | 自动写作主日志 |
| `logs/auto_write_subprocess.log` | 子进程详细输出 |
| `logs/prompts/{project}/` | 每次 LLM 调用的完整 prompt |

## 四、技术架构

### 4.1 入口层

| 入口 | 职责 | 使用场景 |
|------|------|---------|
| `cli.py` | 命令行入口：init / write / state / pipeline | 手动单章/范围写作 |
| `auto_write.py` | 全自动持续写作，含重试、审阅、冷却 | 无人值守批量生产 |
| `generate_cilin_chapters.py` | 基于辞林系统实验性生成前5章 | 文风实验、前5章调优 |

### 4.2 编排层

| 组件 | 职责 |
|------|------|
| `BatchWriter` | 单章/批量入口：构建上下文 → 执行流水线 → 保存文件 → 更新数据库 |
| `WritingPipeline` | 10 阶流水线编排：重试、跳过、校验、调用 Expander |
| `ChapterContextBuilder` | 组装 ChapterContext：大纲、角色、债务、辞林注入等 |
| `InputGovernor` | 编译 Director 任务卡，生成 SceneWriter 输入 |

### 4.3 步骤层（Agent 流水线）

```
Director（任务卡）
  ↓
BeatPlanner（六段式节拍）
  ↓
SceneWriter（正文主体）
  ↓
HookEngineer（章末钩子强化）【字数达标可跳过】
  ↓
DialogueTuner（对话调优）【字数达标可跳过】
  ↓
Polish（润色）【默认关闭】
  ↓
Auditor（审计）【默认关闭】
  ↓
PostWriteValidator + ChapterValidator（本地校验）
  ↓
SpotFix / Expander（修复 or 扩写）
```

> 说明：Polish 和 Auditor 默认关闭，是为控制成本。字数不足时由 Expander 补足。

### 4.4 数据与工具层

| 组件 | 职责 |
|------|------|
| `StateManager` | SQLite 后端，管理跨章状态 |
| `LexiconInjector` | 从 YAML 提取辞林约束，注入 prompt |
| `AntiDetectReviser` | 反检测改写：去 AI 味、句长重排、过渡词替换 |
| `ChapterValidator` | 硬指标校验：字数、他密度、对话占比等 |
| `PostWriteValidator` | 零成本预检：禁用词、模板化开头等 |
| `LLMClient` | 统一 LLM 调用，支持主备 API 和按 Agent 分模型 |

### 4.5 基础设施

- **LLM**：DeepSeek API（`deepseek-chat`）
- **数据库**：SQLite（`world_state.db`）
- **语言**：Python 3.12 + FastAPI（后端 CLI）
- **运行环境**：Windows，Git Bash

## 五、质量保障体系

### 5.1 生成前约束

| 约束来源 | 内容 |
|---------|------|
| 大纲约束 | core_event / face_slap / chapter_hook / emotion_ratio |
| 角色约束 | 对话指纹、肢体语言、情感状态 |
| 世界观约束 | term_dict、技能首次出现章节 |
| 辞林约束 | 场景词条、对话红线、跨角色禁用词 |
| 作者人格 | voice / signature_moves / forbidden_rhetoric |

### 5.2 生成中校验

| 校验器 | 检查项 |
|--------|--------|
| `ChapterValidator` | 字数、他密度、对话占比、连续短句、英文词、精确数字 |
| `PostWriteValidator` | 禁用词、模板化开头、情绪标签、概括性时间 |

### 5.3 生成后处理

| 处理 | 作用 |
|------|------|
| `AntiDetectReviser` | 降低 AI 痕迹分数 |
| `Expander` | 字数不足时扩写 |
| `SpotFix` | 局部修复（当前多被跳过） |

## 七、业务优化方向

### 7.1 短期（本周）

1. **修复标题不一致**
   - `batch_writer.py` 保存文件名时统一使用 `outline.title`
   - 清理历史重复文件和标题错配

2. **稳定批量续写**
   - 从第115章启动 `auto_write.py` 写到第200章
   - 监控 `auto_write_subprocess.log`，确认无新误判

3. **降低意象重复**
   - 扩展"废动作池"，增加 10-15 个可选意象
   - 在 prompt 中增加"禁止连续两章使用同一意象"规则

### 7.2 中期（本月）

1. **重建前 30 章（可选）**
   - 用修复后的完整流程重写前30章，作为质量标杆
   - 重点修复：角色一致性、伏笔埋设、节奏控制

2. **完善 chapter_metrics**
   - 让 IWR、平台分、DNA 匹配真正写入数据库
   - 增加质量趋势看板

3. **优化反检测改写**
   - 将 `AntiDetectReviser` 从后处理改为轻量修正
   - 前置约束（辞林注入）已做，后续降低改写强度

### 7.3 长期（产品化）

1. **多项目支持**
   - 当前 `world_state.db` 已支持 project_id，但工具和流程默认单项目
   - 抽象为可复用的 Novel-OS CLI

2. **成本与质量平衡**
   - 当前 Polish/Auditor 默认关闭，可在关键章节启用
   - 引入"章节质量分档"，高价值章节用更多 LLM 预算

3. **人工审核接口**
   - 增加"待精修"章节导出
   - 支持人工标注 → 反馈优化 prompt

## 九、一句话总结

> Novel-OS 当前是一个**以数据库状态为中心的 LLM 顺序写作流水线**。数据库修复后，系统终于具备了真正的跨章记忆能力。下一步最优先的是**修复标题不一致**，然后**稳定续写第115-200章**，同时逐步提升前30章质量作为发布标杆。
